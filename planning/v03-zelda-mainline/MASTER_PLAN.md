# Storyteller 0.3 × Zelda Mainline：全盘规划

## 0. 文档性质

这是一份开工前的总蓝图。它负责回答：研究什么、如何证明研究完整、如何把案例变成可调用模式、Skill 如何选择而不是照搬、如何接入现有工作流、怎样验证“有味道但不是复制”、失败后回到哪里，以及如何保证旧版本不被破坏。

这份文档不宣称任何具体《塞尔达传说》作品事实，也不构成正式蒸馏成果。作品边界和事实必须在下一阶段通过可追踪来源建立。

## 1. 项目目标与非目标

### 1.1 最终目标

建立一套可长期扩展、可审计、可检索、可版本锁定的游戏设计知识系统，并由 0.3 总导演自主完成以下闭环：

1. 读取任意剧本，先建立不受参考作品污染的故事模型；
2. 从故事矛盾、玩家体验、空间条件、视角、预算和运行时能力中提取查询信号；
3. 从已发布设计库检索多个候选模式，解释选择与拒绝理由；
4. 选择一个贯穿作品的核心机制家族和最多三个辅助家族；
5. 将抽象原理原创化为该剧本独有的玩法、空间、NPC 互动和演出；
6. 编译为可玩的游戏，验证剧情安全、系统因果、空间可达、演出节奏、原创性与可靠性；
7. 保存完整设计血缘，使成品可以复现、回滚和继续迭代。

第一套知识 Pack 只覆盖经正式范围清单确认的《塞尔达传说》正传作品，横跨 2D 与 3D。第一轮成品实验使用 0.21 的 2D 俯视像素产品 Profile，但知识库本身不绑定这一视角。

### 1.2 成功不是“资料很多”

项目只有同时满足下列条件才算成功：

- 约定范围内的作品与分析维度均有明确覆盖状态；
- 每条事实、推断和跨作品结论都有可追踪证据；
- 案例已被抽象成可移植的设计不变量、选择条件和实现接口；
- 选择器既会选择，也会拒绝不适合的模式；
- 模式在当前剧本中有教学、发展、组合或反转、综合考核，而不是一次性彩蛋；
- 成品的“味道”来自探索、规则、空间、反馈和自主发现，不来自任天堂专有表达；
- 同条件三臂体验中，总导演与 Zelda Pack 的增量收益能分别被用户感知；
- 每次生成都能由版本锁和内容哈希解释。

### 1.3 非目标

- 不制作《塞尔达》百科全书、攻略合集或剧情资料库；
- 不复制地图、谜题步骤、Boss 阶段、角色、道具外形、台词、音乐、UI 构图或标志性动作；
- 不让“像素、绿衣、宝箱音效”等表层符号冒充设计能力；
- 不要求每个剧本强行使用 Zelda Pack；“不适合，放弃选择”是合法结果；
- 不在浏览器运行时加载研究语料；运行时只读取批准后的游戏生产数据；
- 不覆盖或原地改写 0.1、0.2、0.21。

## 2. 核心架构决策

### 2.1 三个正交变量

0.3 必须把以下内容拆开：

| 变量 | 回答的问题 | 示例 | 版本单位 |
|---|---|---|---|
| Story semantics | 这个故事是什么，谁知道什么，何时揭示 | `story-model.json` | 单次生产产物 |
| Product profile | 游戏以什么形式运行 | 0.21 俯视像素、未来横版或 3D | Profile / Adapter |
| Inspiration pack | 可以借鉴哪些抽象设计知识 | `zelda-mainline` | Pack release |

这样才能做三臂实验：原生 0.21、0.3 但关闭 Pack、0.3 且启用 Zelda Pack。B/C 隔离知识 Pack 的收益，A/B 隔离总导演架构的收益。

### 2.2 知识不写进巨型 Skill

正式目标目录在开工后才创建：

```text
design-library/
  library-manifest.json
  schemas/
  taxonomies/
  packs/
    zelda-mainline/
      pack-manifest.json
      corpus-scope.json
      sources/
      observations/
      claims/
      works/
      patterns/
      relations/
      coverage/
      flavor-profile.json
  core/
    patterns/
    perspective-adapters/
    composition-rules/
  indexes/
  benchmarks/
  governance/
  releases/

.codex/skills/
  curate-game-design-library/
  select-game-design-patterns/
  build-interactive-stage-game-v03/
```

选择根目录 `design-library/` 而非把知识正文放进 `.codex/skills/`，是为了让知识成为与 Codex 实现解耦的、机器可读的项目资产。Skill 只承载维护、检索、组合、原创化和验证流程。

### 2.3 五层隔离

1. **证据层**：逐作事实、观察、来源、版本、定位与置信度。
2. **蒸馏层**：去除作品表层表达的通用模式、组合规则和反例。
3. **选择与原创化层**：从当前剧本查询、评分、拒绝、组合与变形。
4. **视角与运行时适配层**：把抽象模式翻译到 2D 俯视、横版、2.5D 或 3D 能力。
5. **评估与回归层**：验证覆盖、血缘、剧情、原创性、可玩性、品味与复现。

生成游戏时默认只读取索引和被选中的通用模式；不直接把逐作详细观察送入生成上下文。

### 2.4 数据格式与生成文档

- JSON / JSONL 是唯一权威数据；
- Markdown 档案、索引页和统计报告由权威数据生成；
- 稳定 ID 永不复用；废弃条目保留 `deprecatedBy` / `supersedes`；
- 每个发行版冻结 Schema、Pack、模式、适配器和内容哈希；
- 所有关系均使用受控词表，不用模糊自然语言代替引用。

## 3. 《塞尔达》研究范围规则

### 3.1 G0 前不得凭印象填写“全正传”

正式范围必须先建立 `corpus-scope.json`，逐项记录纳入、排除或待审理由及来源。它还必须保存权威目录快照、快照内容哈希、快照之间的差异，以及候选全集与本地登记的双向对账。Frozen 状态的 `review` 数必须为 0。若权威来源本身仍有冲突，只能宣称“已覆盖 scope-vX 的已裁决集合”，不能宣称“全正传完成”。

### 3.2 建议的范围政策

- 以可引用的官方系列归类和作品资料为主要依据；
- 2D、2.5D、3D 是分析标签，不是互斥知识孤岛；
- 移植版默认归入同一个 `workId`，保存版本差异，不重复计算作品覆盖；
- 重制或机制显著变化的版本建立 `variantId` 与差异档案，不自动当作全新作品；
- 扩展内容、多人作品、边界作品进入 `scope-review`，由明确规则裁决；
- 外传默认不纳入第一版 Pack，但必须出现在排除表并写理由，防止无意遗漏；
- 后续官方分类变化通过新 Scope 版本处理，不静默改写旧发行版；
- Candidate-universe completeness 的分母来自冻结的权威快照并集，不来自研究者“已经发现”的条目数。

### 3.3 每个作品的最低分析维度

所有纳入作品使用同一套受控维度，允许 `not_applicable`，但不允许无解释空白：

1. 玩家幻想与体验承诺；
2. 核心循环与节奏单位；
3. 玩家动词与组合方式；
4. 世界状态、规则与因果；
5. 拓扑、地标与空间记忆；
6. 门控、捷径、回访与旧空间重释；
7. 谜题语法与信息结构；
8. 地城或封闭挑战结构；
9. 探索、奖励与好奇心管理；
10. 成长、资源和经济；
11. 战斗、敌人和 Boss 考核；
12. NPC、任务与社会反馈；
13. 叙事、揭示和演出；
14. 摄像机、走位和视觉引导；
15. 教学、提示、反馈与难度曲线；
16. UI、输入和可读性；
17. 失败、恢复、无障碍与防软锁；
18. 视角相关实现约束；
19. 硬件、联机或多人结构；
20. 生产成本与运行时能力。

## 4. 来源、事实与推断

### 4.1 来源等级

| 等级 | 类型 | 主要用途 |
|---|---|---|
| A | 游戏本体、官方手册、官方页面、官方开发资料 | 作品事实与规则确认 |
| B | 开发者访谈、公开演讲、技术资料、第一方档案 | 设计意图与实现背景 |
| C | 学术研究、专业设计分析、可信出版物 | 解释、比较与外部框架 |
| D | 社区资料、Wiki、论坛、视频分析 | 发现线索；不能单独支撑关键结论 |

所有外部记录都要包含来源 ID、标题、发布者、URL 或馆藏信息、访问日期、精确定位、支持哪些 claim、置信度和权利说明。

来源等级由 `kind + 发行主体 + 可复查性` 的矩阵计算，不允许分析者把社区资料手填为 A。关键 claim 必须至少有一个 A/B 来源；事实 claim 与解释/比较 claim 使用不同引用字段。反向“某来源支持哪些 claim”的列表由 Claims ledger 生成，避免双向手填漂移。

### 4.2 Claims Ledger

每个陈述必须明确属于：

- `fact`：可由来源直接支持；
- `interpretation`：分析者对一个或多个事实的解释；
- `comparison`：不同作品或版本的对照；
- `lineage`：有证据支持的演变关系；
- `analogy`：结构相似但不宣称历史影响。

没有直接证据时只能使用 `analogousTo`，不能写 `evolvedFrom`。冲突来源必须并存并进入待审状态。

### 4.3 版权与原创性操作边界

- 默认保存转述和结构化观察，不保存攻略正文、长引文、地图、截图、音乐和资产；
- 如确需短引文，受来源许可和项目引用上限约束，并记录用途；
- 通用模式不得依赖角色名、道具名、地点名或专有世界观才能成立；
- 单一作品独有观察先标记 `single-title`，不能直接进入自动选择；
- 模式进入发布候选前，必须通过“换世界观、换对象、换拓扑、换反馈、换剧情因果后是否仍成立”的压力测试；
- 本规则是工程上的原创性边界，不替代法律意见。

## 5. 从作品到可调用模式的蒸馏管线

### 5.1 数据层级

```text
作品范围清单
  → 原子观察（Observation）
  → 作品档案（Title Dossier）
  → 跨作品候选族（Pattern Family）
  → 通用设计模式（Released Pattern）
  → 视角/运行时适配器
  → 选择索引与组合规则
  → 0.3 生产调用
```

逐作研究阶段不急着写“普遍规律”。只有证据覆盖通过后才允许跨作品聚类，避免先有结论再找案例。

### 5.2 原子观察

每条观察至少记录：

- 发生在哪个作品、版本和上下文；
- 前置状态、玩家输入、状态转移、反馈和后果；
- 事实描述与分析解释分别是什么；
- 对应哪些分析维度；
- 证据定位和置信度；
- 它是可移植候选、作品特例还是暂缓判断。

观察的粒度应小到可以被多个模式引用，也应完整到能解释因果，而不是“这个谜题很好玩”一类评价。

### 5.3 可调用模式的最低内容

一条 Released Pattern 必须回答：

1. 它解决什么设计问题，追求什么玩家体验；
2. 什么故事信号、空间条件和玩家能力表明它适合；
3. 什么情况下必须禁用或放弃；
4. 它依赖哪些玩家动词和世界状态；
5. 行动如何造成玩家可感知的持久后果；
6. 如何经历教学、练习、变化、组合或反转、综合考核；
7. 如何改变地图、NPC、演出、UI、声音和镜头；
8. 需要哪些运行时能力，成本和降级路线是什么；
9. 与哪些模式协同、冲突、依赖或互斥；
10. 如何处理失败、恢复、提示和无障碍；
11. 哪些原理可保留，哪些表层细节严禁迁移；
12. 哪些观察、作品和反例支撑它，置信度如何；
13. 自动测试和人工体验应看到什么证据。

### 5.4 模式种类

- 原子机制；
- 交互道具；
- 世界状态规则；
- 空间、门控与回环；
- 谜题与信息结构；
- 探索与奖励；
- 战斗与 Boss；
- NPC 与任务；
- 教学、提示与反馈；
- 演出、镜头与场景切换；
- 节奏和宏观结构；
- 多模式编排方案。

### 5.5 模式调用接口

每条模式至少暴露五个类型化接口，不能只写自然语言说明：

- `detect`：用受控 operator、字段路径和类型，从故事及产品约束发现信号；
- `score`：声明 criterion、权重、证据字段和 hard veto；
- `instantiate`：声明参数类型、从哪个上游字段绑定、必填性和变形约束；
- `emit`：把 effect primitive 与约束输出到玩法、演出、空间、编译和验收的具体 contract/field；
- `validate`：用 fixture、证据路径、比较符和预期值定义机器断言。

Effect primitive 只描述通用状态事务，如设置事实、改变地图层、解锁路径、移动演员、改变 NPC 状态、触发演出 Cue、建立检查点。它不得包含来源作品专有对象。演出和空间 handoff 还必须类型化表达 reveal gate、NPC tactic、interaction window、actor blocking、镜头/场景切换、skip/replay、导航不变量和无障碍要求。

## 6. 选择器设计

### 6.1 检索流程

```text
生产章程 + 故事模型 + Product Profile + Runtime Capabilities
  → 提取故事信号与体验目标
  → 查询分面索引
  → 运行时、预算、禁用条件预过滤
  → 读取少量候选详情
  → 独立评分与拒绝理由
  → 组合冲突检查
  → 原创化映射
  → 输出候选与约束，不越权改写故事
```

选择器不得把完整证据库塞进上下文，也不得只凭关键词匹配。例如剧本出现“钥匙”不代表必须选择锁钥机制。

### 6.2 建议评分模型

在硬门禁通过后，以 5 分制评分：

| 维度 | 权重 |
|---|---:|
| 与故事矛盾和主题的相关性 | 25% |
| 能否承载揭示、角色行动和演出 | 20% |
| 可重复发展与组合深度 | 15% |
| 世界状态因果清晰度 | 10% |
| 空间记忆、回访与旧信息重释 | 10% |
| 运行时能力匹配 | 10% |
| 制作预算匹配 | 5% |
| 原创化余量 | 5% |

剧情事实冲突、运行时不可实现且无替代、明显复制风险、无法恢复的软锁风险均为否决项，不由加权平均抵消。

### 6.3 组合预算

- 默认 1 个核心机制家族；
- 最多 3 个辅助机制家族；
- 核心机制必须跨多个幕或区域重复出现并发生质变；
- 至少一个应用要让旧地图、旧物件或旧信息获得新含义；
- 最终段落必须综合此前真实操作过的规则；
- 一次性小游戏必须有不可替代的剧情功能，否则删除；
- 演出、空间和玩法必须服务同一个体验主轴；
- 没有合适组合时，输出 `abstained: true`，回到无 Pack 的原创设计流程。

### 6.4 设计血缘

最终采用模式需记录：

- 使用的模式 ID 与版本；
- 对应故事、场景和体验目标；
- 保留的抽象原理；
- 删除的来源表层细节；
- 对当前世界观、对象、拓扑、反馈和因果的改写；
- 相对来源结构指纹在拓扑图、对象角色图、动作序列、反馈签名和叙事功能上的差异；
- 运行时映射；
- 在最终产物中可检查的证据。

被拒候选也要保存理由，才能诊断选择器而不是只看幸运结果。

## 7. 0.3 Skill 体系

### 7.1 `curate-game-design-library`

维护期 Skill，不参与每次游戏生成。职责：

- 管理 Scope、来源、Claims、逐作覆盖和版本差异；
- 校准不同分析者的观察粒度；
- 跨作品聚类、去重、处理反例和矛盾；
- 管理模式状态、关系、原创性审核和发行；
- 生成索引、覆盖报告、迁移记录和变更日志。

### 7.2 `select-game-design-patterns`

生产期专业 Skill。职责：

- 从故事模型和运行时提取检索信号；
- 加载锁定版本的索引及有限候选；
- 打分、拒绝、组合、原创化和能力协商；
- 输出 `pattern-recommendations.json`；
- 不改写 canon，不替玩法、演出和关卡部门做最终决定。

### 7.3 `build-interactive-stage-game-v03`

用户唯一直接调用的 0.3 总导演。职责：

- 建立生产章程并锁定 Library、Pack、Profile 和专业 Skill；
- 保证故事分析先于模式检索；
- 把候选分别交给玩法、演出、空间部门，并统一成一条主轴；
- 调用美术、编译、路线模拟和浏览器 QA；
- 运行 Zelda flavor 与 clone-risk 两个独立门禁；
- 输出验收报告、设计血缘和完整版本锁。

### 7.4 保护旧版本

- 0.1、0.2、0.21 与现有专业 Skill 首先按本规划的基线哈希冻结；
- 0.3 通过新增可选 handoff 输入调用现有专业 Skill；
- 若共享 Skill 必须改变语义，创建 0.3 专用适配 Skill 或版本化副本，不直接修改旧行为；
- 不把无版本的 `build-interactive-stage-game` 别名改指 0.3；
- 旧游戏不读取新库，旧锁文件不自动升级。

## 8. 0.3 生产阶段与产物所有权

| 阶段 | Owner | 权威产物 | 上游变化后的失效范围 |
|---|---|---|---|
| 00 Brief | 0.3 总导演 | `production-charter.json` | 全部 |
| 05 Lock | 0.3 总导演 | `library-lock.json` | 15 及下游 |
| 10 Story | 故事分析 | `story-model.json` | 15 及下游 |
| 15 Patterns | 模式选择 | `pattern-recommendations.json` | 20 及下游 |
| 20 Gameplay | 玩法设计 | `gameplay-design.json` | 30/40/50/60/报告 |
| 30 Performance | 演出导演 | `performance-plan.json` | 40/50/60/报告 |
| 40 World | 场景关卡 | `stage-plan.json` | 50/60/报告 |
| 50 Art | 美术导演 | 美术圣经、资产清单 | 60/视觉报告 |

## 9. “塞尔达味道”的可检验定义

第一版 flavor profile 不用专有视觉符号，而用设计行为定义：

1. 玩家因好奇和地标主动选择方向；
2. 环境规则一致，答案可由观察和实验发现；
3. 新工具、知识或状态改变旧空间的可行动性；
4. 路径形成回环、捷径和有意义的回访；
5. 教学优先通过安全情境和反馈完成；
6. 同一规则经历变化、组合、反转和综合考核；
7. 奖励既确认发现，也扩展未来可能性；
8. 玩家感觉结论由自己获得，而不是被文本宣告；
9. 叙事、NPC 和演出响应玩家造成的世界变化；
10. 失败可理解、可恢复，不以信息不透明制造难度。

这只是待研究验证的初始假设。正式 flavor profile 必须由跨作品证据和反例修订。

原创性另设 clone-risk gate。高 flavor 分不能抵消对具体原作关卡或表达的明显复刻。来源结构指纹只保存在 Evidence/审核侧，不进入生成上下文；由未参与实例化的 reviewer 做盲源归因，若能稳定猜出某个具体原作关卡则直接否决。

同时引入少量只用于评价、不可进入可调用库的非 Zelda 冒险设计对照。模式需区分 `zelda-discriminative`、`adventure-generic` 与 `uncertain`，防止把任何优秀冒险游戏的通用品质都误称为“塞尔达味”。

## 10. 研究与生产组织

### 10.1 角色

| 角色 | 单一责任 |
|---|---|
| 总负责人 | 冻结范围、裁决边界、批准 Gate 与发行 |
| Corpus steward | 作品清单、版本关系、覆盖矩阵 |
| Source auditor | 来源等级、定位、Claims 与冲突 |
| Title analyst | 单作原子观察和档案，不做跨作定论 |
| Pattern distiller | 跨作聚类、反例、抽象和关系 |
| Originality reviewer | 表层迁移、相似性和改写压力测试 |
| Schema/tool engineer | Schema、索引、验证、迁移和锁文件 |
| Selector designer | 检索、评分、拒绝、组合与基准 |
| Runtime adapter owner | 视角与能力映射、降级和成本 |
| Evaluator | 路线、盲测、品味、回归与发布证据 |

一个人或代理可以兼任多个角色，但同一模式的作者不能独自完成最终原创性与发布审核。

### 10.2 并行规则

- 先用少量、差异明显的校准样本统一观察粒度，再并行分析全范围作品；
- 每个 `workId` 同一时刻只有一个写入 owner；
- 来源登记、作品观察和模式候选使用稳定 ID，可独立合并；
- 共享 taxonomy、索引和 released patterns 只通过串行 merge queue 更新；
- 每轮合并运行 Schema、引用、孤儿节点、循环关系、重复与覆盖检查；
- 代理不能自行宣布 Gate 通过，必须提交机器证据和待人工项。

## 11. 分阶段实施路线

### P0：Planning Freeze（本轮）

产出：本目录、契约草案、基线锁、门禁和校验器。

退出条件：`npm run check:v03-plan-freeze` 与 `npm run check:v03-deployment` 通过；正式研究目录和 0.3 Skill 不存在；用户认可总体边界。

### P1：Scope & Contract Freeze

工作：用官方和高质量来源确认全正传范围、版本规则、taxonomy、Schema、来源政策、原创性规则和完成度算法。

退出条件：G0 通过；所有 contract 有 owner、版本和变更机制；研究者可以在不自行发明字段的情况下写入数据。

### P2：Calibration Pilot

工作：从冻结范围中选取少量结构差异明显的作品做双人独立观察，比较粒度、证据和维度解释；只建立 observation 与 dossier，不发布通用模式。同时建立一条 `calibration-only` provisional pattern，纵向跑通 typed hooks、参数绑定、effect primitives、四部门 handoff 与输入→预期输出验证。

退出条件：分析者一致性达到约定阈值；Schema 不再频繁破坏性变化；样本 G1/G2 以及纵向切片的样本 G3 通过。纵向切片不计入 Released Pattern 数量。

### P3：Corpus Coverage

工作：按批次覆盖所有纳入作品和必要版本差异；持续生成 coverage dashboard、冲突与缺口列表。

退出条件：标题 100% 有状态；核心维度 100% covered 或有理由的 N/A；一般维度达到门禁；无来源关键事实为 0。

### P4：Cross-title Distillation

工作：聚类 observations，建立候选族，分离不变量和表面实现，补反例、关系、组合与视角差异。

退出条件：每条观察都归入模式、作品特例或有理由 deferred；重复和未解决冲突为 0；Gold patterns 通过原创性审查。

### P5：Library Alpha / Beta

工作：发布版本化 Pack、索引、适配器、组合规则、coverage 和 flavor profile；建立迁移与废弃机制。

退出条件：G3–G5 通过；从 Pattern 可以闭环追到 Observation 和 Source；稳定发行可由哈希复现。

### P6：Skill Integration

工作：正式创建 curator、selector 和 v0.3 director；只读取锁定发行；增加 handoff、失效传播、failure routing 和 Skill 前向测试。

退出条件：旧 Skill 基线未漂移；选择器在正例、反例和多随机种子基准上通过 G6–G9。

### P7：Controlled Game Experiment

工作：用同一《小岛惊魂》剧本、同一 0.21 产品 Profile、相同内容和美术预算生成三组：A 为原生 0.21；B 为 0.3 但关闭所有 Inspiration Pack，并阻止 Zelda 派生 core 模式泄漏；C 为同一 0.3 且只启用锁定版本 Zelda Pack。另用至少一个未参与调参的新剧本 fixture 复测，降低熟悉剧本与重复体验效应。

退出条件：三组都有 parity report，确认资产、运行时、时长和内容预算可比；所有路线可达、无软锁、存档一致；匿名体验不显示版本和“塞尔达”字样；预注册最小可感知收益和行为阈值，收集探索、理解、回访、求助和自主发现数据。

### P8：RC & Stable

工作：修复根因并加入回归；执行用户冷体验、揭盲访谈、原创性审查、复现与回滚演练。

退出条件：全部硬门禁通过；用户认可“有设计味道但不是具体模仿”；冻结 Stable 发行与回滚点。

## 12. 完成度模型

| 等级 | 含义 | 不足以做什么 |
|---|---|---|
| L0 | 范围清单冻结 | 不能声称已研究 |
| L1 | 原子观察与来源覆盖 | 不能声称已蒸馏 |
| L2 | 每部作品档案完成 | 不能声称模式可调用 |
| L3 | 跨作品模式族与反例完成 | 不能接入生产 |
| L4 | Released Pattern、索引、适配器和测试完成 | 不能声称 Skill 有效 |
| L5 | 选择器、盲测、品味与复现通过 | 才能发布 0.3 Stable |

建议统计：

- Candidate-universe completeness = 已登记候选 / 冻结权威快照并集；
- Scope resolution = included 或 excluded 的候选 / 已登记候选，Frozen 必须为 100%；
- Work coverage = 达标作品 / 纳入作品；
- Dimension coverage = covered 或有理由 N/A / 应检查维度；
- Evidence closure = 可追到有效 Source 的 claims / 全部 claims；
- Observation disposition = 已进入 pattern、unique 或 deferred 的 observations / 全部 observations；
- Pattern readiness = 通过 Schema、证据、反例、原创性、运行时与测试的模式 / 候选模式；
- Selection quality = 基准用例中满足预期属性且无硬否决的运行 / 全部运行。

完成度不能只用一个平均数掩盖关键空白。

## 13. 版本、失效与回滚

### 13.1 独立版本

- Orchestrator：`build-interactive-stage-game-v03@0.3.x`；
- Library Schema：独立 major/minor；
- Library content release：独立语义版本；
- Pack：`zelda-mainline@x.y.z`；
- Perspective/runtime adapter：独立版本；
- 各次游戏：保存完整 `library-lock.json`。

### 13.2 版本变化

- 只修正来源、拼写或非行为信息：patch；
- 增加观察、模式、关系、组合规则或兼容适配：minor；
- 改 Schema 语义、选择逻辑或已有模式行为：major；
- ID 永不复用；删除改为 deprecated；迁移必须有 machine-readable mapping。

### 13.3 回滚

- 0.1、0.2、0.21、0.3 并存；
- Stable 前不切换默认别名；
- 回滚通过重新选择上一个 Skill、Pack、Schema 和 Adapter 的完整锁组合；
- 不在原发行目录上覆盖数据，不让新库改变旧游戏；
- 发布前必须用一份旧锁重新生成或验证，证明依赖仍可定位。

## 14. 风险登记

| 风险 | 早期信号 | 控制措施 | Owner |
|---|---|---|---|
| 范围无限扩张 | 正传定义反复变化 | Scope 版本、纳入/排除表、变更审查 | Corpus steward |
| 上下文爆炸 | 每次生成加载全库 | 分面索引、先检索后读详情、候选上限 | Selector owner |
| 案例多但无法调用 | 条目只是作品摘要 | Observation/Pattern 分层、五接口 Gate | Distiller |
| 玩法博物馆 | 大量一次性小游戏 | 1+3 预算、发展曲线、终局综合考核 | Director |
| 表面模仿 | 具体地图或符号高度相似 | 原创化字段、clone-risk 独立审查 | Originality reviewer |
| 词面误选 | 因“钥匙”等词触发模式 | 故事因果信号、多种反例、多种子基准 | Selector owner |
| 破坏剧本 | 玩法提前泄露事实 | Story 先于检索、canon Gate、owner 边界 | Story/Director |
| 引擎做不到 | 选中能力超预算 | 检索预过滤、能力协商、明确替代或失败 | Adapter owner |
| 自动分高但不好玩 | 指标与体验背离 | 冷体验、揭盲访谈、用户品味硬门禁 | Evaluator |
| 旧版本漂移 | 共享 Skill 被修改 | 基线哈希、版本化适配、回归 | Release owner |
| 研究不可审计 | 缺 locator 或 claims | Source Schema、引用闭环、G2 | Source auditor |
| 多代理写冲突 | 共享索引被并发覆盖 | 单 owner、merge queue、自动验证 | Project lead |

## 15. 开工决策点

正式研究开始前只需要用户确认以下方向，而不要求用户参与具体玩法或逐作设计：

1. 接受“知识库 / 选择器 / 总导演 / 产品 Profile”分层；
2. 接受先冻结官方依据的正传范围，而不是立即凭记忆列清单；
3. 接受 Zelda flavor 与 clone-risk 双门禁；
4. 接受用相同 0.21 Profile 做三臂实验，而不是把美术差异误判为玩法收益；
5. 接受 Stable 前由用户本人完成最终冷体验品味验收。

用户不需要手工蒸馏资料。系统可以自主完成搜集、结构化、蒸馏、Skill 编写和游戏生成；用户最有价值的参与点是范围裁决（如出现真正争议）、阶段性品味校准和最终冷体验。
